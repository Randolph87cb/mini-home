# 首版场景验证方案

## 目标

这份文档用于约束 `mini-home` 首版客厅场景的验证方式，避免后续改素材、改布局、改交互时再次出现：

- 只看到客厅一角
- 电视、沙发、茶几等关键物件位置失真
- 互动元素互相遮挡
- 道具拖拽后脱离有效区域

## 验证分层

首版场景验证分四层进行，缺一层都不够稳。

### 1. 页面内置布局校验

当前页面已经内置一套基础布局校验，直接在页面右侧“场景校验”卡片中显示结果。

它会自动检查：

- 关键物件是否大部分可见
- 电视是否仍在左侧
- 信件板是否仍在右上区域
- 单杠是否仍在右侧
- 单杠与信件板是否发生大面积重叠
- 零食和水杯是否还落在茶几区域内
- 当前角色状态图是否被裁掉

规则失败时，先修布局，不要继续叠加功能。

### 2. 调试可视化

点击页面里的“显示校验”按钮后，场景会进入调试模式：

- 显示基础网格
- 给场景元素加虚线边框
- 显示每个资源的 `id`

这个模式用于人工确认：

- 资源点击区是否合理
- 元素是否互相遮挡
- 某个资源是不是因为透明留白过大而看起来位置不对

### 3. 交互冒烟检查

仓库内统一提供：

```powershell
npm run verify:scene
```

这条命令会自动：

- 构建当前前端
- 启动本地预览
- 打开页面并检查“场景校验”卡片是否通过
- 回归默认场景、电视、信件板、单杠、桌面道具拖拽这几条主链路
- 失败时输出错误，并把失败截图落到系统临时目录

自动冒烟通过后，再做一次人工视觉检查。

当前人工视觉检查优先使用截图，而不是只靠页面文案或坐标判断。原因是：

- 人物朝向是否合理
- 电视与人物的正反关系是否合理
- 家具和角色的视觉重心是否自然

这些问题更适合看截图，而不是只看规则是否通过。

### 4. 人工交互检查

每次改完首版场景后，至少人工检查下面五项：

1. 默认打开页面时，能看到完整客厅，而不是只有一个局部角落。
2. 点击电视后，角色切到看电视状态，电视切到亮屏态。
3. 点击信封后，角色切到读信状态，并出现展开信件层。
4. 点击单杠后，角色切到引体状态，并能看出两帧循环。
5. 拖动零食和水杯后，仍然只能落在茶几区域内。

如果这次改动涉及默认摆位、角色朝向、电视位置或编辑模式，请额外截三张图检查：

1. 默认页截图
2. 电视态截图
3. 编辑模式截图

## 当前已知局限

- 目前部分 AI 生成透明素材自带留白偏大。
- 当资源留白过大时，元素虽然通过了“在画面内”的校验，但视觉位置仍可能不够自然。

这类问题不能只靠程序布局解决，通常需要：

- 重新裁边资源，或
- 重新生成更适合场景的单资产

## 后续规则

- 修改 `assets/scene-manifest-v1.json` 后，必须重新看“场景校验”卡片。
- 修改场景资源尺寸、位置、层级后，必须打开一次调试模式。
- 修改场景交互或场景挂载逻辑后，必须运行一次 `npm run verify:scene`。
- 如果新增新的核心交互物件，应把新的布局检查逻辑补进 `app.js`，并同步更新本文件。
